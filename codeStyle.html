0. Syntax

  1. Prefixes are used to differentiate between styles belonging to different entities. Following prefixes are defined:
    • rs: In-house code
      • Syntax: "rs" + ProjectName + remaining style
  2. Child Elements: are contained in various entities. For example, an ordered list has list item(s), therefore the following style forms according to the naming convention: 
    
                                .rsPm-orderedList-listItem {}
                                .rsPm-orderedList-listItemIcon {}
                                .rsPm-orderedList-compact {}
  3. Modifiers: are used to describe the state of an entity or resource. For example, a button can have a disabled state or a hover state and can change the colors depending on state change. Modifiers are used in places where the state of an element is being changed, dynamically.Modifiers are defined using the convention _key_value and the value part can be optional. Therefore, the above button example, can be depicted as: 
    
                                .rsPm-button_opacity_90 {}
  4. Style Examples: For example, we want to give top margin 5% to certain element: 
    
                                .rsPm-topMargin-5 {}
                                .rsPm-button_backgroundColor_lightBlue {}

1. Declaration Order

Define all the styles in alphabetical order, with appropriate comments, so that its quick and easy to find code of other team developers and it acts as a definite place to add new styles. 


                        /* ALIGNMENT STYLES */
                        .rsPm-button-centerAlign {
                            text-align: center;
                        }
                        /* BACKGROUND STYLES */
                        .rsPm-jumbotron-cyanBackground {
                            background-color: #00FFFF;
                        }
                        /* CAROUSEL STYLES */
                        .rsPm-carousel_slidingEffect_fast {
                            ...
                        }

2. Tips

  1. Use link instead of @import.
  2. Media queries should be added along with CSS rules in their declaration order, to keep all the code related to that section in one place.
  3. Single declarations should be declared one per line like multiple declarations. 
    
                                    /* Use */
                                    .rsPm-button_color_white {
                                        color: white;
                                    }
                                    /* Don't Use */
                                    .rsPm-textColor-red { color: red; }
  4. Don't cut down the code by using shorthand notations. 
    
                                    /* Use */
                                    .rsPm-jumbotronMargin {
                                        margin-left: 1%;
                                        margin-right: 1%;
                                    }
                                    /* Don't Use */
                                    .rsPm-jumbotronMargin {
                                        margin: 1% 1% 0 0;
                                    }
  5. Don't nest the code and use seperate lines for it.
  6. Comments are the soul of any piece of code. Ensure that appropriate comments are provided in declaration order and anywhere else required, in capital letters so that they can be differentiated from the rest of the code. Long comments are appreciated instead of short ones. Following example depicts the desired behaviour. 
    
                                    /* ALIGNMENT STYLES */
                                    .rsPm-signUpForm {
                                        /* GIVING ALIGNMENT TO THE FORM WRT TO BACKGROUND IMAGE AND OTHER ELEMENTS ON THE PAGE.*/
                                        ...
                                    }
  7. Class names can be long like rsPm-paginationButton_hover_color as they provide easy debugging and improves readability of the code. Avoid using short class names. 
    
                                    /* Use */
                                    .rsPm-paginationButton_hover_lightColor {}
                                    .rsPm-textAlign-center {}
                                    .rsPm-formButton-textAlign-center {}
                                    /* Don't Use */
                                    .rsPm-pagination {}
                                    .rsPm-textCenter {}
                                    .rsPm-buttonCenter {}
  8. Use class names instead of tag names while defining styles as the browser performance is known to be affected by this.

0. Directory Structure

All folder names should be in simple-lowercase alphabets without camelcase or hyphenation words. 


                        root
                          |-styles
                          |   |-lib
                          |   |  |-css
                          |   |  |  |-CSS FRAMEWORK CSS FILES
                          |   |  |-fonts
                          |   |     |-CSS FRAMEWORK FONT FILES
                          |   |-dev
                          |      |-css
                          |         |-DEVELOPER DEFINED CSS FILES
                          |-scripts
                          |   |-lib
                          |   |  |-js
                          |   |     |-JS FRAMEWORK FILES
                          |   |-dev
                          |      |-controllers
                          |      |     |-DEVELOPER DEFINED JS FILES
                          |      |-services
                          |      |     |-DEVELOPER DEFINED JS FILES
                          |      |-directives
                          |            |-DEVELOPER DEFINED JS FILES
                          |-pages
                              |-USER DEFINED HTML PAGES

0.1. Files

  • Naming convention for HTML, CSS and JS files should be in CamelCase, for example, FileOne.html and not fileOne.html
  • Every HTML page should have its own CSS file, for example, FileOne.html should have FileOne.css and similarly, for other pages. Although, when styles start to overlap among different pages then, one reusableStyles.css file should be created and those styles should be added to it.
  • All images should be saved with img-[number] naming convention, for example, img-1, img-2, etc.

2. Editor Configuration

Best way to ensure that every team member's code looks alike, in terms of indentation, code-markup, styles, etc; is to use a editor configuration tool like EditorConfig. Following is a sample .editorconfig file that can be used for most of the HTML, CSS, JS projects. 


                            root = true
                            [*]
                            charset = utf-8
                            indent_style = space
                            indent_size = 4
                            trim_trailing_whitespace = true
                            end_of_line = lf
                            insert_final_newline = true
